Overviews

Summary

It is highly recommended to look through the general documentation of a construct before proceeding, as there all the general construct structure and preferences are described.

Overviews are constructs that allow an editor to iterate over certain items (folders, pages, files or images) to render information about those items. Use cases for overview tags are for example:

  • Listing news article pages in a certain folder to render a teaser and links to the article.
  • Listing images to render an image gallery.
  • Listing folders to provide direct links on the start page to important parts of the site (e.g. a list of projects).

Construct

An overview construct must contain at least the following construct parts:

  1. An overview part: This part defines which pages are to be iterated over. This part must be editable and should generally be hidden.
  2. A handlebars part: This part will render the actual overview. This part must be neither editable nor hidden. See "Overview Properties" for an description the the available properties in the Handlebars template.

Other construct parts are not needed in most cases.

Selecting entries

The overview part offers two properties to decide which items will be listed by the overview: The type of selectable objects and which objects of that type are to be included in the overview. Please note that, according to the example above, the files for this part should be named part.overview.json and part.overview.ds.json, for the part itself, and its data source accordingly, where overview is a part name.

Object Types

The property "Selectable Objects" of an overview part defines which objects may be selected by the overview. The construct can allow any combination of the following object types to be selected:

  1. Files. The property value for it is FILE.
  2. Folders. The property value for it is FOLDER.
  3. Images. The property value for it is IMAGE.
  4. Pages. The property value for it is PAGE.

When a tag of this type is inserted in a page, only the object types that are defined in the tag type can be selected. When an editor inserts a tag of this type, the property "Displayed Elements" allows to select one of the object types and the overview tag will iterate only over this object type. The Handlebars template is responsible for generating correct markup according to the selected type, for example to render information for pages directly when "Selectable Objects" is set to "Pages" and rendering information from folder start pages when "Selectable Objects" is set to "Folders".

The property name for this in the part.<part_name>.json is listType.

Selection Method

The property "Select Type" of an overview part defines from where the listed objects are taken. The tag type can allow any combination of the following select types:

  1. Automatically from current folder: All items that are in the same folder as the page containing this tag. The property value for it is AUTO.
  2. Folder: All items in the specified folders. The property value for it is FOLDER.
  3. Individually: Select the items to be iterated over individually. The property value for it is MANUAL.

As with "Selectable Objects", an editor can choose only one select type for inserted overview tags.

 The property name for this in the part.<part_name>.json is selectionType.

Other selection settings

Additionally to the "Select Type" an editor can specify a maximum number of entries that the overview should contain and if the overview should recursively add items from folders that are in the scope of the overview. For example, consider a folder called "News" is selected which contains subfolders "May" and "June", which contain itself news pages. When the property "recursive" is checked in the tag, the overview will list pages that are contained directly in the "News" folder, as well as pages contained in the subfolders "May" and "June".

Sorting

How the items of the overview are sorted, is defined by the editor who inserts the tag in a page. The tag fill dialog for an overview contains the settings "Order by", which defines on which item property the items are sorted, and "Order", which defines the sorting direction.

Currently the following options are available for "Order by":

  1. Alphabetically: The items are sorted alphabetically by their name property. The property name to order is name.
  2. Date (created): The items are sorted by their creation date. The property name to order is cdate.
  3. Date (edited): The items are sorted by their edit date. The property name to order is edate.
  4. User defined: The items are sorted in the order the editor defines, and the property name has to be provided by the user.

Examples

Following is an overview construct, with a keyword example_overview and a name [Example] Project Overview, that belongs to the project example_overview_project. As every other construct, it holds a flat structure of several files:

  • Mandatory gentics_structure.json, holding the construct description
  • part.handlebars.hbs with the HTML markup of an overview, powered by Handlebars . Please note that items helper, that is provided by us (see below) is referenced through the project-specific prefix, i.e. example_overview_project.items.
  • part.overview.json with the default values of an overview itself. This file can remain unchanged, yet required for the overview structure. Please note that overview part of file's name belongs to the name of the construct part's keyword, so has to be changed accordingly, if the keyword is changed.
  • part.overview.ds.json contains a data source definition for the overview (as above, with the keyword overview).

Along with this a custom Handlebars helper is provided, filtering out the pages, picked by a user into the overview, by an existence of a status.key custom property. According to the convention, the name of a file and function within match, in our case items.js and function items(). It does not belong to the overview construct, but to the example_overview_project project, and so is placed separately, to be reused from multiple constructs, if required.

Handlebars

Helpers

The Handlebars helpers contain a single helper named "items", which iterates over the elements from the overview and skips all items which do not have a status set in the "example_overview_project" object property.
handlebars/helpers/items.js
function items(items, options) {
    if (items == null || items.size() === 0) {
        return options.inverse();
    }

    const filtered = [];
     items.forEach(function(entry) {
        const op = entry.object.example_project;
        if (op != null
            && op.parts != null
            && op.parts.status != null
            && op.parts.status != null
            && op.parts.status.key
        ) {
            filtered.push(entry);
        }
    });

    if (filtered.length === 0) {
        return options.inverse();
    }

    return options.fn({ items: filtered });
}
