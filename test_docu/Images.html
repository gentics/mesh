Adding images

Basics

It is strongly recommended to look through the basic structure of the Construct first.

In order to add an image to the content, one typically creates a dedicated construct. This construct contains the HTML markup for an image, the image picker, plus it is usual to have an alt text, containing a copyright or a description, so an input for that is provided as well. 

Before we start, we need to answer some questions, and, based on the answers, proceed with an implementation. These are:

  1. What is the name of the project we currently work on?
  2. What is the name of the construct we want to create?
  3. Where its source image is taken from, files at CMS or static resource?
  4. Where its alternative text is taken from, user input or static text?
  5. Do we need a multi-size image?

As an example, let's say the original designed HTML markup for the content images is looking as follows:

<figure title="Author: Max Mustermann">
        <img src="default-example-image.jpg" alt="Author: Max Mustermann" style="max-width: 100%; height: auto;">
</figure>

NB: For sake of simplicity we use a common alt text for both alt text at the <img> and <figure>.

We will answer the questions above as follows:

  1. The project name is my_gentics.
  2. The construct name is example_image.
  3. The image is populated with the image files from CMS.
  4. The alternative text is populated from the user input.
  5. We need no multi-size, for now.

Let's implements this.

As the image source is to be taken from the CMS, the static part has to be be replaced with the reference to the CMS image. In order to do this, we will use Handlebars. According to the construct parts documentation, our image construct is looking like following:

  1. A CMS picker of a Page/Image/File/Folder URL type, set up for picking images. Keyword -- img. Its content (image) is not directly depicted in the resulting content, as we have a custom markup for it, so it is marked as hidden (TODO: Link to the `hidden` option). We limit the picker to the images only. It is also editable (TODO: link to the `editable` option). As per picker documentation, the actual image's URL is accessible through the .url suffix.
  2. An alternative text input. Keyword -- alt. An input of Text type, so its content is accessible through the .text suffix.
  3. A Handlebars part, containing the designed markup and modifications to use the content of inputs above. It will look like this:
part.handlebars.hbs
<figure title="{{cms.tag.parts.alt.text}}">
    <img src="{{cms.tag.parts.img.url}}" alt="{{cms.tag.parts.alt.text}}" style="max-width: 100%; height: auto;">
</figure>

The snippet above shows how we reference the other parts of our construct, img for an image source, alt for the alt text and figure tooltip. The construct is planned to be a sub-tag, but contain no other tags.

Please note that cms.tag.parts. prefix is mandatory to access the parts of the construct we are currently in. More information on the CMS Handlebars functionality can be found here.

Let's play fair with the content author and add a warning of a missing image, when in edit mode:

part.handlebars.hbs
{{#and (not cms.tag.parts.img.url) cms.rendermode.edit}}
    <div>Please fill in the image file in the tag menu.</div>
{{else}}
    <figure data-imageid="{{ cms.tag.parts.img.id }}" data-tagname="{{ cms.tag.name }}" title="{{cms.tag.parts.alt.text}}">
        <img src="{{cms.tag.parts.img.url}}" alt="{{cms.tag.parts.alt.text}}" style="max-width: 100%; height: auto;">
    </figure>
{{/and}}

And it's done!

Adding resizing capabilities

Since the example above works, but is pretty basic in the way how an image is displayed. Let's add a way to resize the image, so it saves us some payload and has flexible size.

This will also mean that we need to change our answer to the question 5 (yes, we do need multi-size now) and 2 (our new construct will have a name example_resizable_image).

Let's say we need four image sizes in total. We can implement this via a Datasource, providing each size as a name+value pair, where a name is human readable, and value is a percentage from the original image width:

  1. Tiny -- 25%
  2. Small -- 50%
  3. Medium -- 75%
  4. Big -- 100%

With this we add a Single Selection part to our construct, both hidden and editable, with the just created datasource. Let's pick a Medium as a default value, so that every image used within this tag has size 75%. 

The resizing itself is done via gtx_gis helper, available at CMS. Let's add it, starting with the fixed width of 825 pixels. Please also note that here we changed the population of an alt text from {{cms.tag.parts.alt.text}} to {{gtx_render cms.tag.parts.alt}}. It is not necessary, yet highly recommended to use gtx_render helper on every user-defined part, since it makes a correct output for each data type the part is using.

part.handlebars.hbs
{{#and (not cms.tag.parts.img.url) cms.rendermode.edit}}
    <div class="message info" style="font-size: 14px; line-height: 1rem;">
          Please fill in the image file in the tag menu.
    </div>
{{else}}
    <figure title="{{gtx_render cms.tag.parts.alt}}">
        <img src="{{{gtx_gis cms.tag.parts.img.target width=825 height="auto" mode="prop"}}}" alt="{{gtx_render cms.tag.parts.alt}}" style="max-width: 100%; height: auto;">
    </figure>
{{/and}}

It does work, but we still have to use the size selector. For that we create an own Handlebars helper, where wrapping a gtx_gis call with a context with all the necessary sizes.

Please pay attention to the syntax of the helper, as Gentics CMS uses not the standard Javascript, but a OpenJDK Nashorn dialect. More info on the CMS Handlebars helpers is available here.
to_gtx_gis_size.js
function to_gtx_gis_size(size, options) {
	let ctx = {
		'width': 825,
		'height': 'auto',
		'mode': 'prop'
	};
	if (size.value === "100%") {
		ctx.width = 1280;
	} else if (size.value === "50%") {
		ctx.width = 550;
	} else if (size.value === "25%") {
		ctx.width = 225;
	}
	return options.fn(ctx);
}

Here we create a ctx variable, containing our context of width, height and mode respectively. The width is corrected according to the value of a size. With options.fn() we make a context-dependent wrap, as shown in the following markup:

part.handlebars.hbs
{{#and (not cms.tag.parts.img.url) cms.rendermode.edit}}
    <div>Please fill in the image file in the tag menu.</div>
{{else}}
    <figure data-imageid="{{ cms.tag.parts.img.id }}" data-tagname="{{ cms.tag.name }}" title="{{gtx_render cms.tag.parts.alt}}">
        {{#my_gentics.to_gtx_gis_size cms.tag.parts.size}}
                <img src="{{{gtx_gis cms.tag.parts.img.target width=this.width height=this.height mode=this.mode}}}" alt="{{gtx_render cms.tag.parts.alt}}" style="max-width: 100%; height: auto;">
        {{/example_resizable_image.to_gtx_gis_size}}
    </figure>
{{/and}}

The gtx_gis helper is called with all the necessary parameters, populated from the wrapping helper's context via this. prefix, e.g. this.width, this.height, this.mode.
